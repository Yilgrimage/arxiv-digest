## 一、429 根因分析

**答案：间隔不完整，确实可能触发 429。**

### 当前 arXiv API 调用序列（按执行顺序）

| # | 调用 | 目标 | 与上一次 arXiv 调用的间隔 |
|---|------|------|---------------------------|
| 1 | `fetch_by_topic("LLM post-training")` | export.arxiv.org | — |
| 2 | `time.sleep(3)` |  | ✓ |
| 3 | `fetch_by_topic("agentic RL")` | export.arxiv.org | 3s |
| 4 | `time.sleep(3)` |  | ✓ |
| 5 | `fetch_by_topic("LLM agent")` | export.arxiv.org | 3s |
| 6 | `fetch_alphaxiv_ids()` | alphaxiv.org | **非 arXiv，不计入** |
| 7 | `fetch_by_ids(alphaxiv_ids)` | export.arxiv.org | **❌ 0s！无 sleep** |
| 8 | `fetch_huggingface_papers()` | huggingface.co | **非 arXiv，不计入** |
| 9 | `fetch_by_ids(hf_ids)` | export.arxiv.org | **❌ 0s！无 sleep** |

**问题**：步骤 5→7、7→9 之间没有 `time.sleep(3)`。虽然 topic 之间守规矩，但 topic 结束后立即连发两个 `fetch_by_ids`，完全没给 arXiv API 喘息时间。

**另外**：`fetch_by_ids` 内部 batch 之间有 sleep(3)，但如果 alphaxiv IDs >20 个，会触发多 batch，这也会导致总请求数增加。

## 二、设计-实现一致性审计

### ✅ 一致的

| 设计目标 | 实现状态 |
|---------|---------|
| 三源聚合 (topics + alphaxiv + huggingface) | ✓ |
| LLM 逐篇评估 + rerank + 过滤 | ✓ |
| 只推送最终报告 | ✓（cron job 已加约束） |
| 按日期分文件保存日报 | ✓ |
| 推荐历史追踪 + 重复高亮 | ✓ |
| 外部热度信号 (HN, citations) | ✓ |
| 论文单篇档案 + 语义检索 | ✓ |
| 增量收集（部分成功可保存） | **部分实现，但有漏洞** |

### ❌ 不一致 / 有漏洞

#### 1. 【严重】`fetch_by_ids` 不在进度跟踪中

**设计理念**：每个数据源独立跟踪，失败可重试，不浪费已有进度。

**实现漏洞**：
- `fetch_source_with_progress` 跟踪的是 `alphaxiv_ids`（从 alphaxiv.org 抓 ID 列表）
- 但它**不跟踪** `fetch_by_ids(ids)`（用这些 ID 去 arXiv API 查详情）
- 如果 `alphaxiv_ids` 成功但 `fetch_by_ids` 失败（429），进度文件标记 `alphaxiv_ids: ok`
- 下次重试时，看到 `ok` 就跳过，**永远不会重试 `fetch_by_ids`**
- 结果：alphaxiv 板块永远为空，即使 IDs 已经拿到了

**同理**：`fetch_huggingface_papers` 内部的 `fetch_by_ids` 也不在跟踪中。

#### 2. 【严重】`fetch_source_with_progress` 的状态判断过于粗糙

```python
status = "ok" if papers else "failed"
```

- `fetch_alphaxiv_ids` 返回的是 `list[str]`（ID 列表），不是论文对象
- 即使返回了 10 个 IDs（status="ok"），后续的 `fetch_by_ids` 可能全部失败
- 进度文件显示 "ok"，但实际论文数据为 0
- 无法区分"数据源确实没新论文"和"网络请求失败"

#### 3. 【中等】huggingface 板块未走 `fetch_source_with_progress`

```python
# 当前代码
papers = fetch_source_with_progress(progress, "huggingface", fetch_huggingface_papers, ...)
```

看起来走了，但 `fetch_huggingface_papers` 内部又调了 `fetch_by_ids`，这个子调用不在跟踪中。

#### 4. 【中等】cron timeout 与重试策略不匹配

- 设计理念：exit code 2 → 等待 15 分钟 → 重试 `--raw`
- 实现：timeout=900s（15 分钟），等于等待时间本身
- 问题：等待 15 分钟后再跑 `--raw` + LLM 评估，几乎必然超时
- 结论：timeout 应至少 30 分钟（1800s）或更长

#### 5. 【轻微】`fetch_url` 对 429 的检测不够 robust

```python
if result.stdout.strip() == "Rate exceeded." or len(result.stdout.strip()) == len("Rate exceeded."):
```

- 假设 arXiv 返回体严格等于 "Rate exceeded."（14 字节）
- 如果返回体带换行、前后空格、或不同措辞，检测失效
- 实际今天看到的就是 `len=14`，所以这部分暂时可用，但脆弱

## 三、修复建议（按优先级）

### P0 - 必须修

1. **统一 arXiv API 调用的 sleep(3) 间隔**
   - 在 `fetch_by_topic` 和 `fetch_by_ids` 之间、以及两个 `fetch_by_ids` 之间插入 sleep(3)
   - 最佳方案：把 `fetch_by_topic` 和 `fetch_by_ids` 统一走一个带速率限制的包装器

2. **把 `fetch_by_ids` 纳入进度跟踪**
   - `alphaxiv` 拆成两个阶段：`alphaxiv_ids` + `alphaxiv_papers`
   - `huggingface` 拆成两个阶段：`huggingface_html` + `huggingface_papers`
   - 每个阶段独立跟踪

### P1 - 应该修

3. **精细化状态判断**
   - 区分 "ok"（有论文数据）、"empty"（正常返回但 0 篇）、"failed"（网络/API 错误）
   - 只有 "ok" 才跳过，"failed" 和 "empty" 都允许重试

4. **增加 cron timeout**
   - 从 900s → 1800s 或更长

### P2 - 可以优化

5. **更 robust 的 429 检测**
   - 用 curl `-w "\n%{http_code}"` 获取真实 HTTP 状态码，而不是猜 body 内容

## 四、当前 arXiv API 状态

- export.arxiv.org 目前从这台机器访问：**间歇性 429 / 连接超时**
- 官方要求：多次调用之间 **3 秒延迟**
- 当前实现：topic 之间满足，但 topic→fetch_by_ids 不满足
